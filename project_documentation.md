# Housing Society Management System - Project Documentation

This document outlines the detailed logic, architecture, and functionality implemented in the Housing Society Management System project. The project is a client-side web application built using HTML, CSS, and Vanilla JavaScript, demonstrating a fully functional frontend with data persistence through the browser's `localStorage`.

## 1. Core Architecture & Data Storage
- **Local Storage Database:** The application functions entirely without a backend server. All CRUD (Create, Read, Update, Delete) operations read from and write to the browser's `localStorage`.
- **Data Seeding:** Upon the first load, the application checks if data keys (like `hsm-owners`, `hsm-maintenance`, `hsm-expenses`) exist. If they do not, it populates the storage with default dummy data (arrays of objects).
- **Theme Management:** The platform supports both Light and Dark modes. It respects the user's OS preference (`prefers-color-scheme`) by default but allows manual toggling. The user's preference is saved in `localStorage` under the key `hsm-theme` so it persists across page reloads.

## 2. Authentication & Routing (`Auth.html`)
- **Role-Based Redirection:** The authentication page acts as a router. When users navigate to `Auth.html` from the landing page, URL query parameters (e.g., `?role=admin` or `?role=resident`) are parsed by JavaScript. The script automatically switches the UI to the correct context.
- **Dummy Login Flow:** As a demonstration environment, form validation has been bypassed. Clicking "Log In" or "Create Account" instantly checks the selected role and uses `window.location.href` to redirect the user to either `dashboard.html` (for Admins) or `owner-dashboard.html` (for Residents). No actual backend validation occurs.

## 3. Landing Page (`index.html`)
- **Marketing & Navigation:** The landing page serves as the entry point, showcasing features and pricing.
- **Dynamic Links:** The "Login as Admin", "Login as Resident", and "Get Started" buttons dynamically route visitors to the `Auth.html` page, appending the appropriate `?role=` query parameter to preset the login state.

## 4. Admin Dashboard (`dashboard.html`)
The Admin Dashboard provides complete control over society records and finances.
- **Global Dashboard Statistics:** Reads from `localStorage` to compute and display total collected funds, current month expenses, pending dues, and total units.
- **Visual Analytics:** Features a custom SVG donut chart generated by JavaScript to break down expenses by category (e.g., Housekeeping, Utilities, Repairs).
- **Resident Management:** Admins can add, edit, and delete resident profiles. These changes instantly reflect in the `owners` array in `localStorage`.
- **Expense Tracking:** Allows admins to record new expenses with categories, amounts, and dates.
- **Notice Board Management:** Admins can publish society announcements, flagging them with priority levels (Urgent, Informational, Normal).
- **Payment Verification Workflow:** 
  - When residents make a payment, it appears in the "Recent Payments" table with a **"Pending"** status.
  - An **Approve** button allows the admin to verify the transaction.
  - Approving a payment runs logic that locates the corresponding unit in the `owners` and `maintenance` arrays, marks their statuses as `ok` (Paid), and updates the transaction status, instantly clearing the resident's dues.
- **System Settings:** Admins can update global society variables (Society Name, Total Wings, Total Units). This section also features a "Reset Data" danger button that wipes `localStorage` and restores the original dummy arrays.

## 5. Resident Dashboard (`owner-dashboard.html`)
The Resident Dashboard provides a restricted, read-only view of core society data, tailored to a specific resident.
- **Unit Simulator:** Because there is no real login session, a dropdown selector allows the user to switch between different units to simulate what different residents would see. Changing the unit filters the dataset and re-renders the DOM for that specific owner.
- **Financial Transparency:** Residents can see the overall society financials (Total Collected vs. Total Expenses) to ensure transparency.
- **Billing & Payments:** 
  - Residents see their current outstanding maintenance dues.
  - **UPI Payment Simulation:** If a resident clicks "Pay Now" and selects **UPI**, the form submission is intercepted. A dummy "Scan to Pay" modal containing a fake QR code is displayed.
  - **Payment Submission:** Once the resident clicks "Simulate Payment Success," JavaScript creates a transaction record in `localStorage` with a `status` of `"pending"`. The resident's dashboard updates to show "Pending Admin Verification," and the payment button is disabled to prevent duplicate submissions.
- **Notices View:** Renders the society notices published by the admin in a read-only list.
- **Profile Management:** Residents can view their unit details (which are locked and managed by the admin) but have the freedom to update their personal contact information (Phone and Email). These updates are saved to a separate `resident-contacts` object in `localStorage`.
